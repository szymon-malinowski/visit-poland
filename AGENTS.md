# AGENTS.md

## Project overview

Visit Poland is a responsive travel discovery application built with React 19,
TypeScript, Vite, TanStack Router, Tailwind CSS 4, and DaisyUI 5. Read
`SPEC.md` before making changes; it is the source of truth for product,
architecture, styling, routing, and accessibility requirements.

## Development commands

Run commands from the repository root:

```bash
npm install       # install dependencies
npm run dev       # start the Vite development server
npm run lint      # lint the repository
npm run build     # type-check and create a production build
npm run preview   # preview the production build
```

There is currently no automated test script. Before considering a change
complete, run both `npm run lint` and `npm run build`.

## Repository structure

```text
src/
├── components/
│   ├── atoms/
│   ├── molecules/
│   ├── organisms/
│   └── templates/
├── pages/
├── routes/
├── index.css
├── main.tsx
└── routeTree.gen.ts
public/images/
```

Follow the Atomic Design dependency direction:

```text
atoms -> molecules -> organisms -> templates -> pages -> routes
```

Lower-level components must not import from higher-level layers. Put a new
component in the smallest layer that accurately describes its responsibility.
Pages own route-specific data and composition; templates own reusable layouts.

## Implementation conventions

- Use TypeScript and functional React components.
- Match the formatting and import style of the file being edited. Do not create
  unrelated formatting churn.
- Keep route files in `src/routes` limited to route configuration and page
  imports. Put route-level UI in `src/pages`.
- Never edit `src/routeTree.gen.ts` manually. It is generated by the TanStack
  Router Vite plugin from the files in `src/routes`.
- Reuse existing components and collection patterns before adding abstractions.
- Keep page-specific content and data out of generic templates.
- Store static visual assets under `public/images` in an appropriate subfolder
  and use descriptive file names and alt text.

## Styling

- Use DaisyUI primitives such as `btn`, `card`, `navbar`, `menu`, `badge`,
  `list`, `hero`, and `footer` whenever they fit the UI.
- Use Tailwind utilities for layout, spacing, typography, responsiveness, and
  small visual adjustments.
- Use semantic DaisyUI tokens such as `primary`, `secondary`, `base-100`,
  `base-200`, `base-300`, and `base-content` instead of hard-coded colors where
  possible.
- Keep all global styling in `src/index.css`. Do not add component stylesheets
  or bespoke CSS selectors unless DaisyUI and Tailwind cannot reasonably express
  the requirement.
- Preserve the red-and-white Polish visual identity and verify both light and
  dark themes.
- Design from a 320 px mobile width upward and check responsive behavior at
  mobile and desktop sizes.

## Accessibility and UX

- Use semantic landmarks and appropriate heading order.
- Ensure navigation communicates the active route.
- Give informative images meaningful alt text; hide decorative visuals from
  assistive technology.
- Maintain sufficient contrast and visible keyboard focus states.
- Use native interactive elements and labels. Do not replace buttons or links
  with non-semantic clickable containers.
- Confirm that text, controls, and layouts do not overflow or overlap at narrow
  widths.

## Change discipline

- Keep changes scoped to the request and preserve unrelated work in the
  worktree.
- Do not commit generated build output such as `dist/`.
- Update `SPEC.md` when intentionally changing documented product behavior or
  architectural conventions.
- For UI changes, manually inspect the affected routes in the browser in both
  themes and at representative mobile and desktop widths.
- In the final summary, state what changed and report the results of lint and
  build checks, including any check that could not be run.
